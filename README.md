# PINN (Task & Model Execution System)

PINN — это распределенная система для управления жизненным циклом моделей и выполнения вычислительных задач в изолированных Docker-контейнерах. Проект позволяет собирать образы из исходного кода, планировать выполнение задач с ограничениями по ресурсам (CPU, RAM, GPU) и кэшировать результаты вычислений.

## Основные возможности

*   **Build-as-a-Service**: Сборка Docker-образов из `tar.gz` архивов с потоковой передачей логов сборки клиенту.
*   **Изолированное выполнение**: Запуск задач в контейнерах с жесткими лимитами ресурсов.
*   **Умное планирование**: Поддержка отложенного запуска задач (`scheduled_at`).
*   **Кэширование**: Автоматическое использование результатов предыдущих запусков при совпадении сигнатуры задачи (ModelID + Input + Envs + Cmd).
*   **S3-совместимое хранилище**: Интеграция с MinIO для хранения входных данных и результатов.
*   **Мониторинг ресурсов**: Отслеживание нагрузки на хост-систему в реальном времени.

## Технологический стек

*   **Язык**: Go 1.25+
*   **База данных**: PostgreSQL (pgx, sqlc)
*   **Очередь и Worker**: Собственная реализация на каналах Go и семафорах.
*   **Контейнеризация**: Docker SDK для Go.
*   **Хранилище**: MinIO (S3 SDK).
*   **Роутинг**: go-chi.

## Быстрый запуск

### Требования
*   Docker & Docker Compose
*   Go 1.25 (для локальной сборки)

### Запуск инфраструктуры
```bash
docker-compose up
```
Это запустит PostgreSQL и MinIO, необходимые для работы сервера.

### Настройка окружения
Создайте `.env` файл на основе примера:
```env
DB_URL=postgres://user:pass@localhost:5432/pinn?sslmode=disable
MINIO_ENDPOINT=localhost:9000
MINIO_ACCESS_KEY=minioadmin
MINIO_SECRET_KEY=minioadmin
PORT=:8080
```

### Запуск сервера
```bash
go run cmd/server/main.go
```

## API Documentation

### Управление моделями (`/model`)

#### 1. Сборка модели из исходников
**POST** `/model/build/`
Собирает Docker-образ из архива и регистрирует модель. Поддерживает Real-time стриминг логов.

*   **Content-Type**: `multipart/form-data`
*   **Части запроса**:
    *   `model`: JSON-метаданные `{"id": "my-model"}`
    *   `artifacts`: Файл `artifacts.tar.gz` (должен содержать `Dockerfile`)
*   **Пример**:
    ```bash
    curl -N -X POST http://localhost:8080/model/build/ \
      -F 'model={"id":"mock"};type=application/json' \
      -F 'artifacts=@artifacts.tar.gz'
    ```

#### 2. Список моделей
**GET** `/model/`
Возвращает список всех зарегистрированных моделей.

#### 3. Удаление модели
**DELETE** `/model/{id}`
Удаляет запись из БД и связанный Docker-образ (если он был собран системой).

### Выполнение задач (`/task`)

ВАЖНО! cpu_limit измеряется в процентах от одного логического ядра. То есть 50 = 50% ядра, 250 = 250% ядра, то есть 2.5 ядра.
Система сама распределяет нагрузку по ядрам, даже в случае cpu_limit < 100.
memory_limit измеряется в мегабайтах.

#### 1. Запуск новой задачи
**POST** `/task/run`
Создает задачу, загружает входные данные и ставит её в очередь на выполнение.

*   **Content-Type**: `multipart/form-data`
*   **Части запроса**:
    *   `metadata`: JSON конфигурация (см. ниже)
    *   `file`: Входной файл данных для модели.
*   **Параметры Metadata**:
    ```json
    {
      "model_id": "mock",
      "container_cmd": ["python", "main.py"],
      "cpu_limit": 50,
      "memory_limit": 512,
      "gpu_enabled": false,
      "timeout_sec": 3600,
      "scheduled_at": "2026-03-20T15:00:00Z" 
    }
    ```

#### 2. Статус задачи
**GET** `/task/{id}/status`
Возвращает текущее состояние задачи, логи ошибок (если есть) и ссылку на результат (если готова).

#### 3. Остановка задачи
**POST** `/task/{id}/stop`
Принудительно останавливает контейнер и помечает задачу как `stopped`.

### Системные эндпоинты

*   **GET** `/health`: Проверка работоспособности (БД, Docker, Storage).
*   **GET** `/stats`: Метрики системы (CPU load, RAM usage, количество активных контейнеров).

## Структура проекта

*   `cmd/server`: Точка входа в приложение.
*   `internal/domain`: Доменные модели и интерфейсы.
*   `internal/service`: Бизнес-логика (TaskService, ModelService).
*   `internal/docker`: Взаимодействие с Docker API.
*   `internal/repository`: Работа с PostgreSQL (через sqlc).
*   `internal/storage`: Интеграция с MinIO/S3.
*   `migrations`: SQL миграции для схемы базы данных.
